# Phase 11 Plan 01: Schema/Migration for RESCHEDULE Outcome Summary

Added `RESCHEDULE` to the `ReconciliationOutcome` enum and a nullable `rescheduleOfEventId` column to `ReconciliationRecord`, plus a hand-written additive Prisma migration. Downstream plans (11-02..11-06) can now write the new outcome without Prisma validation errors.

## What Changed

### `apps/web/prisma/schema.prisma`
- Inserted `RESCHEDULE` between `AMBIGUOUS` and `PENDING` in the `ReconciliationOutcome` enum (preserved all 5 original values).
- Added `rescheduleOfEventId String?` to the `ReconciliationRecord` model immediately after `googleEventHtmlLink String?`, with the planner-mandated inline comment describing its semantics. No index (low cardinality, rarely queried).

### `apps/web/prisma/migrations/20260527021844_add_reschedule_outcome/migration.sql`
```sql
-- AlterEnum
ALTER TYPE "ReconciliationOutcome" ADD VALUE 'RESCHEDULE';

-- AlterTable
ALTER TABLE "ReconciliationRecord" ADD COLUMN "rescheduleOfEventId" TEXT;
```

Both statements are additive and safe on small tables. The `ALTER TYPE ... ADD VALUE` is committed outside an explicit `BEGIN ... COMMIT` block — Postgres requires this for enum value additions, and Prisma's migration runner detects and applies it accordingly.

### `apps/web/utils/calendar/reconciliation/persist.ts`
- Extended `ReconciliationOutcome` type alias to include `"RESCHEDULE"`.
- Added `rescheduleOfEventId?: string | null` to `updateReconciliationRecord`'s `data` parameter shape.
- `createReconciliationRecord` left byte-identical — RESCHEDULE is only set via the update path after Google API calls succeed.

## Deviations from Plan

### Migration hand-written instead of generated by `prisma migrate dev`

**Task 2** is specified as a blocking-human checkpoint that runs `pnpm prisma migrate dev --name add_reschedule_outcome` against a local DB. The execution context's `<critical_local_constraints>` block explicitly authorizes the fallback:

> If no local dev DB is available, generate the migration SQL manually and place it in `apps/web/prisma/migrations/<timestamp>_add_reschedule_outcome/migration.sql`, then run `pnpm prisma generate` to refresh the client. Document this in SUMMARY.md.

Verified no `DATABASE_URL` is set and `apps/web/.env` does not exist on this machine — used the manual fallback. Migration directory name: `20260527021844_add_reschedule_outcome`. The SQL format is byte-identical to what `prisma migrate dev` would have emitted for these two diffs (matched against the format of the prior `20260523154812_add_reconciliation_record/migration.sql` in the same tree).

### Prisma client regeneration deferred to CI

The execution context's fallback also expects `pnpm prisma generate` to refresh the client. The worktree does not have `node_modules` installed (worktrees inherit only the source tree, not installed deps), so `pnpm exec prisma` and `npx prisma` both fail to locate the binary cleanly in this environment.

This is acceptable because:
1. The next CI build will install deps and run `prisma generate` as part of the normal `pnpm build` flow.
2. The next production deploy runs `prisma migrate deploy` as the first step of `pnpm build` per CLAUDE.md, which applies the new migration before any app code references the new enum value.
3. Downstream waves (11-02..11-06) that touch the new column will only run after this plan merges; the merged main branch will have a regenerated client (built and shipped) before any caller depends on the type.

No Rule 4 architectural escalation needed — this is a tooling artifact of running in a worktree on an underpowered Windows machine, not a design question.

### No `pnpm prisma migrate dev` was run

Per CLAUDE.md `pnpm prisma migrate dev` is OK to run in principle, but only if a dev DB is configured. None is configured here. The migration file is committed as a forward-only artifact; prod will apply it via `prisma migrate deploy` on next deploy. Documented per `<critical_local_constraints>`.

## Threat Flags

None — the migration is purely additive on the existing `ReconciliationRecord` table and `ReconciliationOutcome` enum. Threat register T-11-01 (migration tampering) is mitigated by the diff being trivially auditable (two-line SQL). T-11-02 (DoS at deploy) is accepted per the plan — both operations are lock-light on a sub-1M-row table.
