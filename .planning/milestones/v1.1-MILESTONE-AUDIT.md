## CRITICAL — OPS-03 prompt caching is failing in production

Confirmed 2026-06-01 against the Anthropic Console: **zero cache usage over 7 days** ("You're not using prompt caching"). This is the entire goal of Phase 8.5, and Phase 9/11 inherited the same broken pattern.

**Root cause (documented in the code itself):** All three caching paths — classifier (`ai-choose-rule.ts`), reconciliation extract (`extract-prompt.ts`), arbitration (`arbitrate.ts`) — run on the **economy/Haiku** tier (`getModel(user, "economy")`). Every prompt was deliberately sized to ~1500 tokens against a **1024-token cache floor**:
- `extract-prompt.ts:8-11` — "MUST exceed 1024 tokens... ~1500 tokens — well above the floor"
- `arbitrate.ts:26` — "padded past the 1024-token Anthropic cache floor"

But **1024 is the Opus/Sonnet minimum. Haiku's minimum cacheable prefix is 2048 tokens.** A ~1500-token prefix on Haiku is below the floor, so Anthropic accepts `cache_control`, declines to cache, and bills full input price. The AI-SPEC pitfall the team coded against applied the wrong tier's number.

**Current impact:** Inert, not harmful — because nothing caches, there's no 1.25× cache-write premium either. Cost today is exactly as if caching weren't wired in. OPS-03 is simply unmet.

**Why a naive fix may not help:** Anthropic caching only pays off when a second call reuses the prefix within the **5-minute TTL**. At ~85 emails/day (~1 per 17 min) and 1–3 calendar emails/day, extract/arbitration calls are almost always lone cache-*writes* with no read — padding them past 2048 would add the 1.25× write premium for ~zero read benefit, i.e. *increase* cost. The **classifier** is the only path frequent/bursty enough (Gmail can deliver batched history items; shared prefix across all emails) to plausibly benefit from a proper 2048-token cache. Even there the absolute saving is small — the whole pipeline is Haiku, already ~$1/MTok input.

**Options (operator decision — cost/benefit, not a pure code fix):**
1. **Accept + clean up** — caching is the wrong tool at this volume. Remove the inert `cache_control` from extract/arbitrate (and optionally the classifier), delete the misleading "1024-token floor" comments, and reframe OPS-03 as "not viable at single-user volume." Lowest risk, honest.
2. **Fix the classifier only** — pad the classifier system prefix past 2048 tokens, leave extract/arbitrate uncached. Measure cache_read on the Console after a few days; keep only if reads are non-trivial.
3. **Leave as-is** — accept the inert code and the console nag; OPS-03 stays formally unmet.

Recommendation: **Option 1 or 2.** Do not blanket-pad all three paths — that risks raising cost.
